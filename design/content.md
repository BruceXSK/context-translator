# content

每页一份的 content script：持有本页 session 与 pending 上下文，负责悬停翻译、段落抽取、嵌入式译文块与三态显隐、划线悬浮窗、Shadow DOM UI、流式渲染与原位错误反馈。

## Specs

- CT-001 [DONE] hover trigger: pressing the configurable trigger key (default Alt) over a paragraph requests its translation; the trigger is suppressed inside editable elements.
- CT-002 [DONE] paragraph extraction: the translation unit is the text of the block/paragraph containing the cursor.
- CT-003 [DONE] embedded block translation with toggle: the first trigger fetches and inserts a translation block below the paragraph; later triggers toggle its visibility from cache without re-requesting.
- CT-004 [DONE] translation cache: completed translations are cached per paragraph so re-showing never re-requests.
- CT-005 [DONE] selection floating panel: a single reusable, draggable, closable panel near the selection shows the translation; each invocation is a fresh request with no caching.
- CT-006 [DONE] shadow-DOM UI isolation: the selection floating panel lives in a shadow DOM to isolate it from page styles; the embedded hover translation renders in the page DOM (no shadow) so it inherits the page's inline-element styling (links, code) and font.
- CT-007 [DONE] progressive streaming render: the translation UI updates as tokens arrive.
- CT-008 [DONE] session ownership: the content script holds the page's session message array and pending-context buffer.
- CT-009 [DONE] in-place error feedback: failures render in the translation area; the selection floating panel provides a retry control, and the embedded hover block retries by re-pressing the trigger key.
- CT-010 [DONE] source-matched inline translation: the embedded hover translation renders directly after the source text within the same paragraph, with no container box, background, border, or label; its typography (font, size, weight, style, color, line-height, spacing, alignment) inherits the source paragraph's styling via the page's CSS, and it is rendered slightly lighter than the source via a transparency effect, so it reads as a soft continuation of the page rather than an injected widget.
- CT-011 [DONE] inline-format-preserving translation: the hover translation preserves the source paragraph's inline elements; each inline element is replaced with a placeholder that keeps its real tag plus a data-ct-id attribute (no other attributes — href/class/data-*/style are not sent), and a shallow clone of the original (tag + attributes) is stored by id; the placeholder skeleton (text + real-tag placeholders, no original attributes) is sent to the LLM with a preserve-markup instruction (keep translated text inside each element, move elements as units); the returned HTML is sanitized (dangerous tags/handlers stripped) and each tagged element is swapped back for a clone of its original element (event handlers and dangerous URI schemes stripped) with recursively reconstructed translated children; if an original is missing or the output unsafe, the affected span falls back to plain text.
- CT-012 [DONE] hover loading placeholder: while a hover translation is in flight (before the first token), the embedded block shows a "Translating" placeholder followed by an animated braille spinner (10 frames ⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏, ~80ms each) at the position the translation will occupy; it is replaced by the streamed text as soon as the first chunk arrives, and cleared on done/error. The animation is driven by a timer (not CSS), since the block is light-DOM with no shadow (CT-010).
- CT-013 [DONE] line-clamp container compat: when a hovered paragraph's translation block would be clipped by an ancestor carrying `-webkit-line-clamp` (e.g. Google search snippets limited to N lines), showing the translation sets that ancestor's `-webkit-line-clamp` to `none` so the translation is visible; hiding restores the original value (re-set the recorded inline value, or remove the override so the class rule re-applies). The block itself stays `display:block` below the source. Mechanism confirmed against immersive-translate, which sets the same ancestor's clamp to `none`.
- CT-014 [DONE] no duplicate translation block: re-hovering an already-translated region (including after a `-webkit-line-clamp` container expands and shifts layout, or when a `-webkit-box` ancestor isn't matched as a block) toggles the existing block rather than creating a second one. `findParagraph` re-targets the existing `context-translator-block`'s paragraph (its `parentElement`) when a candidate block already contains a prior translation.
- CT-015 [DONE] skip same-language hover: when the hovered paragraph's text is already in the target language, the hover trigger does not send a translation request — no token spend, no session pollution, no DeepSeek cache-prefix disruption (DS-001). Detection runs client-side in `doToggle` before `toggleHover` as a pure Unicode-script ratio test on the paragraph's plain text. The `[lang]` attribute is NOT consulted — `closest('[lang]')` can climb to a page-level tag that does not describe the paragraph (e.g. GitHub's `<html lang="en">` over a Chinese README, which wrongly forced translation of Chinese), and same-script languages cannot be reliably told apart by `lang` anyway. The content's script must match the target language's primary script: for `zh-*`, Han > 20% of judgeable characters with zero kana/hangul (the low threshold lets CJK-heavy mixed Chinese/English text still skip; zero kana/hangul avoids ja/ko false hits); `ja` needs kana > 0; `ko` needs hangul > 30%; `ru`/`uk`/`be`/`bg` need Cyrillic > 50%; `ar` needs Arabic > 50%; Latin-family targets (en/fr/de/es/…) need Latin > 50% with no CJK. An unknown target language never skips. The skip is silent — no UI feedback. Applies to hover translation only; selection-翻译 always sends (explicit user intent). Gated by CFG-004 (default on, not exposed in the settings UI in v1). Known limit: same-script language pairs (Traditional vs Simplified Chinese; en vs fr vs de) are not distinguished — all are skipped by target script, so e.g. hovering Traditional Chinese with a Simplified target skips (no Traditional→Simplified conversion).
- CT-016 [DONE] re-translate button on hover block: after a hover translation completes, a small always-visible ↻ icon button appears at the end of the embedded translation. Pressing it (click or keyboard Enter/Space) re-translates the same paragraph: it re-sends the paragraph's skeleton to the LLM as a re-translate request (SES-006 — via the normal translate path with a `re-translate` marker leading the `<user-instruction>` block, folding pending context/instruction, appended to the committed prefix so the cache prefix stays stable; pending is consumed on commit) and renders the new result in place of the old one. The re-translate bypasses the CT-015 same-language skip (explicit user intent). On error the old translation is cleared and the error is shown in place; the ↻ button stays visible for retry. The button is light-DOM with inline styles (no shadow, same constraint as the spinner) and the paragraph's skeleton is cached in ParaState at first translation so a re-translate does not re-extract the (possibly mutated) DOM.
- CT-017 [DONE] trigger key on a text selection: when a non-empty text selection is active, the trigger key opens the selection floating panel (selectionTranslate, CT-005) instead of hover translation; it is explicit user intent and bypasses the CT-015 same-language skip. The selection covers both document selections and selections inside <input>/<textarea> (window.getSelection() misses the latter, so activeElement.selectionStart/End is read), so removing the right-click "翻译" menu item causes no regression for editable selections.
- CT-018 [DONE] serialized single-flight translation: at most one translate/compress stream is in flight per page; a request that arrives while another is in flight waits for the prior to commit (or error), then builds its message array on the then-current session and streams. With one in flight the single in-flight slot (pendingUserContent) can never be clobbered — eliminating the concurrency corruption where one request's user message is committed with another's assistant response. Fresh translation, re-translate (CT-016), and compress share this single exit. Re-triggering a paragraph whose translation is already in flight or queued is a no-op (no duplicate queued translation).

## Hover translation flow

1. 监听 keydown（触发键，默认 Alt）：若当前存在非空文本选区，改走选区翻译面板（CT-017）；否则取当前鼠标位置所在段落，光标在 `input`/`textarea`/`contenteditable` 内时忽略。
2. 首次触发：抽取段落文本 → 经 session 模块组装消息 → 发给 background 流式翻译 → 在段落下方插入译文块（light-DOM，继承页面内联样式与字体，CT-010/CT-011），逐 token 填充 → 完成后缓存译文、提交响应进 session。
3. 后续触发：切换译文块显/隐，使用缓存，不再请求、不改 session（纯软件行为）。

## Selection translation flow

存在文本选区时按触发键（CT-017）→ 取选区文本 → 组装消息（无缓存）→ background 流式翻译 → 单个可复用悬浮面板（近选区、可拖动、可关闭）渐进显示 → 完成后提交响应进 session。右键菜单不再含「翻译」项，选区翻译统一由触发键承担（document 选区 + input/textarea 选区）。

## Add-instruction flow (no selection)

右键「Add instruction」（无选区时显示，BG-003）→ 弹出内联输入面板（Shadow DOM、textarea + 提交，回车提交 / Shift+回车换行 / Esc 关闭）→ 用户输入补充指引（术语/背景/语气）→ 提交后面板短暂显示「✓ 已加入上下文，下次翻译生效」再收起，文本加入本页 pending 上下文（instruction 类，SES-002）。pending 在下一次**全新翻译**（悬停未译过的段落、或选区 + 触发键）时折入 `<user-instruction>` 块并清空；重显已缓存的译文（toggle）与 ↻ 重翻译不消费 pending（SES-006），故加入指令后需做一次全新翻译才能看到效果。

## UI isolation

悬停译文块为 light-DOM（无 shadow，CT-010），以继承页面内联样式与字体；划线悬浮面板与「Add instruction」输入面板挂 Shadow DOM，样式自包含，避免被宿主页 CSS 影响也避免污染页面。
